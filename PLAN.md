# PLAN.md — Telegram Bot Simulator & DevTools

> Спецификация для агента (Codex). Работай **помилестонно**, сверху вниз. После каждого
> милестона: `go vet ./...` + `go test ./...` должны проходить, затем коммит. Не переходи
> к следующему милестону, пока acceptance-критерии текущего не выполнены. Не добавляй
> зависимости без явной причины (см. раздел «Зависимости»). Не реализуй то, что в «Non-goals».

---

## 1. Что строим (суть в одном абзаце)

Локальный «фейковый Telegram» для разработчиков ботов. Это HTTP-сервер на Go, который:
1. **притворяется Telegram Bot API** — бот направляет свой `baseApiUrl` на нас вместо
   `https://api.telegram.org`, и ничего не замечает;
2. даёт **веб-клиент** (чат в браузере), где можно писать боту и жать кнопки без телефона
   и без туннелей;
3. даёт **«рентген»-панель** — живой поток того, что происходит у бота внутри: какой апдейт
   пришёл, какие методы бот вызвал в ответ, статусы, латентность, ошибки.

**Главный принцип:** работаем на уровне HTTP-протокола Bot API, а не внутри конкретной
библиотеки. Поэтому инструмент **framework- и language-agnostic** — подходит любому боту
на любом языке. Это не библиотека-обёртка, это инструмент рядом с ботом.

## 2. Non-goals (СТРОГО не делать в рамках этого плана)

- ❌ НЕ реализовывать весь Bot API. Только методы из §6. Для остального — generic-заглушка
  (валидный `{"ok":true,...}`), чтобы бот не падал.
- ❌ НЕ подключаться к настоящему Telegram. Мы его заменяем, а не проксируем (proxy-режим —
  отдельный будущий милестон M7, в MVP его НЕТ).
- ❌ НЕ требовать внешних сервисов: ни Redis, ни Postgres, ни Docker как зависимость рантайма.
  Состояние — в памяти. Персистентность (M6) — только встроенный SQLite, опционально.
- ❌ НЕ делать аккаунты/роли/мультитенант. Единственная защита — токен доступа в remote-режиме.
- ❌ НЕ тащить WebSocket. Поток сервер→браузер — через **SSE**. Действия браузер→сервер —
  обычный HTTP POST.
- ❌ НЕ авто-открывать браузер, НЕ предполагать наличие дисплея — инструмент должен жить
  на headless-сервере по SSH так же, как на ноутбуке.
- ❌ НЕ гнаться за идеальной корреляцией трейсов. Цель — честная эвристика (см. §8),
  явно помеченная как «inferred».

## 3. Стек и обоснование

| Слой | Выбор | Почему |
|---|---|---|
| Backend | **Go 1.23+** | I/O-bound нагрузка, один статический бинарник, ~20 МБ RAM, едет на слабом VPS |
| HTTP-роутинг | **stdlib `net/http`** (enhanced `ServeMux`, Go 1.22+) | method+path паттерны есть в stdlib, фреймворк не нужен |
| Логи | **stdlib `log/slog`** (JSON) | структурные логи без зависимостей |
| Стриминг в UI | **SSE** (`text/event-stream`) | однонаправленный поток, авто-реконнект, проще WS |
| Персистентность (M6) | **`modernc.org/sqlite`** (pure Go, без cgo) | один файл, ноль внешних сервисов, кросс-компиляция без C |
| Ротация лог-файла (опц.) | **`gopkg.in/natefinch/lumberjack.v2`** | чтобы логи не забили диск на маленьком боксе |
| Frontend | **Vite + React + TypeScript** | собирается в статику |
| Доставка фронта | **`go:embed`** | фронт вшит в бинарник → один файл, никакого Node в проде |
| Интеграционный тест | **`github.com/mymmrac/telego`** (dev-зависимость) | реальный бот, направленный на наш мок |

**Не Rust** — выигрыш в перформансе тут не в узком месте, цена — медленная разработка.
**Не Python** — тяжелее по памяти, слабее как «лёгкий бинарник на VPS».
**Не Next.js для фронта** — нам нужна статика под `go:embed`, а не отдельный Node-процесс.

## 5. Поток данных (как всё соединяется)

```
[Браузер: чат-клиент]                         [Браузер: рентген-панель]
        │ POST /_sim/inject (message/callback)          ▲ SSE /_sim/events
        ▼                                               │
┌─────────────────────────── Go HTTP server ───────────────────────────┐
│  sim (control-plane) ── enqueue Update ──► store (update queue)       │
│                                              │                        │
│  botapi /bot<token>/getUpdates  ◄── long-poll достаёт Update ─────────┤
│  botapi /bot<token>/<sendXxx>   ── бот шлёт ответ ──► store(messages)  │
│                                              │                        │
│  trace: каждому Update — id; вызовы бота привязываются (§8) ──► ring   │
│  events: новые messages и trace-апдейты ──► broadcast по SSE          │
└───────────────────────────────────────────────────────────────────────┘
        ▲ getUpdates / webhook POST
        │
   [Разрабатываемый бот: baseApiUrl = http://host:port]
```

Цикл: пользователь в браузере пишет боту → control-plane собирает `Update` и кладёт в очередь →
бот забирает через `getUpdates` (или мы шлём webhook-POST) → бот вызывает `sendMessage` и т.п. →
мы сохраняем сообщения и пишем трейс → пушим по SSE → браузер показывает ответ бота в чате,
а панель — трейс.

## 6. Поверхность Bot API (что реализуем)

Базовый путь — как у настоящего Telegram: `(/bot<TOKEN>/<method>`, метод и GET, и POST,
параметры из query **и** из JSON-body **и** из `multipart/form-data`. Ответ всегда обёрнут:
`{"ok":true,"result":...}` или `{"ok":false,"error_code":N,"description":"..."}`.

**Реализовать полноценно:**
- `getMe` → вернуть фиктивного бота (id/username из токена-конфига).
- `getUpdates` → **long-poll**: параметры `offset`, `timeout`, `limit`. Блокируемся до
  появления апдейта или таймаута, отдаём батч, двигаем offset. Это также **flush-граница**
  для корреляции (§8).
- `setWebhook` / `deleteWebhook` / `getWebhookInfo` → переключение в webhook-режим:
  при наличии webhook мы POST-им апдейты на URL бота вместо отдачи через getUpdates.
- `sendMessage` → создать `Message`, сохранить, разослать по SSE. Поддержать `reply_markup`
  (inline + reply keyboard), `parse_mode`, `reply_to_message_id`.
- `editMessageText` / `editMessageReplyMarkup` → мутировать существующее сообщение.
- `deleteMessage`.
- `answerCallbackQuery` → закрыть «часики» на кнопке, опц. показать тост в UI.
- `sendPhoto` / `sendDocument` → принять `multipart`, сохранить байты в памяти, отдавать
  обратно через `/_sim/file/<id>`; в чате показать как медиа.

**Generic-fallback:** любой нереализованный метод → `200 {"ok":true,"result":true}` +
залогировать как `unimplemented` в трейс (чтобы было видно, что бот вызвал что-то, чего мы
не моделируем), но НЕ роняем бота.

**Control-plane (НЕ Bot API, префикс `/_sim/`):**
- `POST /_sim/inject` — тело описывает действие пользователя: `text message` или
  `callback_query` (нажатие inline-кнопки). Сервер строит валидный `Update` и кладёт в очередь.
- `GET  /_sim/state` — текущие чаты/сообщения (для гидрации UI при загрузке).
- `GET  /_sim/events` — SSE-поток (§9).
- `GET  /_sim/file/<id>` — отдать сохранённые байты медиа.
- `POST /_sim/reset` — очистить состояние и буфер.

## 7. Модели данных

Хранить структуры, совместимые с Telegram (чтобы `Update`, который мы отдаём боту, проходил
парсинг любой библиотеки). Минимум полей:

```go
type Update struct {
    UpdateID      int64          `json:"update_id"`
    Message       *Message       `json:"message,omitempty"`
    CallbackQuery *CallbackQuery `json:"callback_query,omitempty"`
}
type Message struct {
    MessageID   int64        `json:"message_id"`
    From        *User        `json:"from,omitempty"`
    Chat        Chat         `json:"chat"`
    Date        int64        `json:"date"`
    Text        string       `json:"text,omitempty"`
    ReplyMarkup *InlineKeyboardMarkup `json:"reply_markup,omitempty"`
    // photo/document — минимально
}
type CallbackQuery struct {
    ID      string   `json:"id"`
    From    User     `json:"from"`
    Message *Message `json:"message,omitempty"`
    Data    string   `json:"data,omitempty"`
}
// User, Chat, InlineKeyboardMarkup, InlineKeyboardButton — по минимуму полей.
```

**Модель трейса (это сам продукт, не побочка):**

```go
type Trace struct {
    ID        string        // корреляционный id
    Inbound   InboundEvent  // апдейт, который пришёл боту (тип, chat_id, краткое содержимое)
    Calls     []OutboundCall// что бот вызвал в ответ
    StartedAt time.Time
    Status    string        // "ok" | "error" | "open"
}
type OutboundCall struct {
    Method     string
    Params     map[string]any // обрезать большие тела
    HTTPStatus int
    OK         bool
    ErrorCode  int
    ErrorDesc  string
    LatencyMS  int64
    At         time.Time
}
```

## 8. Корреляция трейсов (САМАЯ ТОНКАЯ ЧАСТЬ — читать внимательно)

Проблема: когда бот вызывает `sendMessage`, в HTTP-запросе есть `chat_id`, но **нет**
`update_id`, на который он отвечает. Связать «этот вызов — ответ на апдейт N» напрямую нельзя.
Решаем эвристикой и честно помечаем результат как inferred.

**Правила корреляции (реализовать ровно так):**

1. Когда апдейт **доставлен боту** (вошёл в ответ `getUpdates` ИЛИ отправлен webhook-POST) —
   открываем «активное окно» обработки, ключ = `chat_id` апдейта, и создаём `Trace{Status:"open"}`.
2. Любой исходящий вызов бота с тем же `chat_id` в пределах окна **привязывается** к этому трейсу
   (добавляется в `Calls`).
3. **Граница окна (flush):**
   - long-poll: следующий вызов `getUpdates` от бота закрывает все открытые окна — всё, что бот
     сделал между получением апдейтов и следующим поллом, относится к их обработке. Это сильный
     сигнал, используем его как основной для long-poll.
   - webhook: окно закрывается по ответу на webhook-POST **или** по таймауту (напр. 5s).
   - в любом случае — таймаут-страховка, чтобы окна не висели вечно.
4. Вызов с `chat_id`, не совпавшим ни с одним открытым окном (или вовсе без `chat_id`) →
   трейс `Inbound: nil`, помечаем `orphan` — тоже показываем, не теряем.
5. На каждом трейсе и каждом привязанном вызове ставим флаг `correlation: "inferred"` —
   UI показывает это явно, чтобы никто не принимал эвристику за точную истину.

Перфектную корреляцию (например, через инструментацию самого бота) — НЕ делать, это non-goal.

## 9. SSE-контракт

Эндпоинт `GET /_sim/events`, `Content-Type: text/event-stream`. События (поле `event:` +
JSON в `data:`):

- `message` — новое/изменённое/удалённое сообщение (для чат-клиента). Поле `op: created|edited|deleted`.
- `trace` — создан/обновлён/закрыт трейс (для рентген-панели). Поле `op: open|update|close`.
- `ping` — keepalive раз в ~15s.

Реализация: `events.Hub` с регистрацией подписчиков (буферизированные каналы), broadcast,
снятие отвалившихся. Все мутации store и trace проходят через Hub.

## 10. Ограничение ресурсов (обязательно с самого начала)

- Состояние — обычные `map` под `sync.RWMutex` (или actor-через-канал, если так чище).
- **Ring buffer** на трейсы/события: хранить последние N (дефолт **1000**), старое вытесняется.
  Память ограничена по определению — рост невозможен. Это критично для слабого VPS.
- Тела параметров в `OutboundCall.Params` и медиа-байты — обрезать/лимитировать
  (напр. param-значения > 4 КБ усекать, медиа > заданного лимита не хранить целиком).

## 11. Режимы запуска, конфиг, безопасность

Конфиг через флаги + env (env переопределяет дефолты, флаг — env). Три явных пресета `--mode`:

| Режим | Bind | Токен доступа | TLS | Назначение |
|---|---|---|---|---|
| `local` (default) | `127.0.0.1:8080` | нет | нет | ноутбук, всё на localhost |
| `remote` | `0.0.0.0:8080` | **обязателен** | опц. | бот/LLM на VPS, панель открываешь с ноута |
| `proxy` | — | — | — | **M7, не в MVP**: инспектор живого бота |

Параметры: `--mode`, `--addr`, `--token`, `--bot-token` (токен фейкового бота для пути
`/bot<token>/`), `--buffer-size`, `--persist` (путь к SQLite-файлу; пусто = только память),
`--log-format` (json|text), `--log-file` (пусто = stdout).

**Жёсткие правила безопасности:**
- В `remote` без `--token` → отказ старта с понятной ошибкой.
- Токен проверяется middleware на всех `/_sim/*` и на UI; на `/bot<token>/*` — нет (туда ходит
  бот, его «аутентификация» — сам токен в пути).
- В README явно рекомендовать прятать `remote` за Tailscale или Cloudflare Tunnel вместо
  публичного порта; TLS тогда снимает туннель/Caddy. Самим с сертификатами в MVP не возиться.

## 12. Логирование (ДВА разных вида — не смешивать)

**Вид 1 — ops-логи самого инструмента** (это твоя кухня, не показывается пользователю в UI):
- `log/slog`, JSON, уровни. Без zap/zerolog.
- **Recover-middleware** вокруг хендлеров: паника → 500 + стектрейс в лог, сервер живёт.
- Вывод в stdout (дефолт; ротацию отдать systemd/journald) или в файл через `lumberjack`.

**Вид 2 — трейсы бота** (это сам продукт, выпячивается в UI):
- Каждое взаимодействие на границе мок-API → `Trace`/`OutboundCall` (§7), в ring buffer, в SSE.
- Ошибки бота (кривой запрос, 4xx, таймаут, паника на колбэке, `unimplemented`-метод) —
  показываются в панели красным. НЕ путать с ops-логами.

## 14. Милестоны (выполнять по порядку)

### M0 — Скелет
Go-модуль, `cmd/sim/main.go`, config (флаги/env, три режима), slog, recover-middleware,
`GET /healthz`, пустой `go:embed` плейсхолдер фронта, Makefile.
**Done:** `make build` даёт один бинарник; `./sim` отвечает 200 на `/healthz`; логи в JSON;
`remote` без токена не стартует.

### M1 — Минимальный Bot API + long-poll
`store` (chats/messages/update-queue за интерфейсом), `getMe`, `getUpdates` (long-poll),
`sendMessage`, `POST /_sim/inject` (text). Интеграционный тест: telego-бот в процессе,
направленный на мок.
**Done:** инжектишь `/start` → бот получает апдейт через getUpdates → его `sendMessage`
сохранён; тест зелёный.

### M2 — Трейсы + SSE
Корреляция (§8), ring buffer, `trace`-модель, `GET /_sim/events` (SSE), broadcast мутаций.
**Done:** один инжект → SSE отдаёт `trace open` + привязанные `OutboundCall` под одним id +
латентность; ошибочный вызов помечен `error`; всё помечено `inferred`.

### M3 — Фронт: чат-клиент
Vite+React+TS, экран чата (список чатов/сообщений, поле ввода, рендер inline/reply-клавиатур
кликабельными кнопками; клик по inline → инжект `callback_query`). Подписка на SSE `message`.
Сборка → `web/dist` → `go:embed`.
**Done:** в браузере можно полноценно общаться с ботом без терминала и телефона.

### M4 — Фронт: рентген-панель
Живой поток трейсов из SSE; на каждый апдейт — раскрываемая карточка (inbound → список вызовов
→ статус/латентность); ошибки красным; пометка `inferred`; уважение к лимиту буфера.
**Done:** работает демо-сценарий из §15 (тот самый, что записывается на GIF).

### M5 — Расширение Bot API + webhook
`editMessageText`, `editMessageReplyMarkup`, `deleteMessage`, `answerCallbackQuery`,
`sendPhoto`/`sendDocument` (multipart → байты → `/_sim/file/<id>`), generic-fallback.
Webhook-режим: `setWebhook`/`deleteWebhook`/`getWebhookInfo` + реальный POST апдейтов на URL бота.
**Done:** клавиатуры/правки/колбэки/медиа работают end-to-end; webhook-бот тоже работает.

### M6 — Персистентность + снапшот-тесты
`persist` через `modernc.org/sqlite` (за интерфейсом; память остаётся дефолтом). Запись сессии
(последовательность апдейтов + ответы бота) и реплей. Харнесс снапшот-тестов диалогов для CI.
**Done:** записал сессию → реплейнул в тесте без сети → совпало.

### M7 — (Будущее, НЕ в MVP) proxy-инспектор
MITM между живым ботом и настоящим Telegram, та же панель на проде. Спроектировать так, чтобы
`store`/`trace`/`events`/SSE переиспользовались. Сейчас только не мешать этому архитектурно.

## 15. Definition of Done для MVP (демо-сценарий)

Один бинарник + один браузер, без телефона, без туннеля, без настоящего Telegram:
1. Запускаю `./sim` (режим local).
2. Запускаю простого telego-бота с `baseApiUrl = http://127.0.0.1:8080`.
3. Открываю браузер: слева чат, справа рентген-панель.
